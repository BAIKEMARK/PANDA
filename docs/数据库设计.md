# 数据库设计

这种设计方式允许 AI 直接生成建表 SQL 或 Python 数据类，而不需要处理复杂的外键约束，非常适合快速迭代。
---
### 📦 数据库设计原则
1.  **无物理外键**：表与表之间通过 `ID` 字段在逻辑上关联，不在数据库层面强制约束。
2.  **JSON 友好**：大量使用 `JSON` 或 `TEXT` 字段存储动态配置（如 Prompt、评分细节），减少表的数量。
3.  **ID 统一**：全部使用字符串 ID（如 `uuid` 或 `cuid`），方便分布式和前端直接处理。

### 🚀 给 Vibe Coding 的 AI 提示词
你可以直接复制以下 Prompt 给 Cursor/Copilot/ChatGPT，让它生成对应的 `schema.sql` 或 Prisma 文件：
> **Prompt:**
> "我正在开发一个 Python + MySQL 的医疗培训 MVP。请根据以下设计思路，生成完整的 SQL 建表语句：
> 1. 不要使用物理外键，所有关联通过 ID 字段逻辑处理。
> 2. ID 字段统一使用 CHAR(36) 存储 UUID。
> 3. 使用 JSON 类型存储动态数据（如评分详情、消息元数据）。
>
> **表结构需求：**
> 1. **users**: 存 id, email, password_hash, name, role (student/admin)。
> 2. **courses**: 存 id, title, content_url。
> 3. **user_progress**: 关联 user_id 和 course_id，存 is_completed。
> 4. **scenarios**: 存 id, title, system_prompt (大模型提示词), patient_background。
> 5. **chat_sessions**: 关联 user_id 和 scenario_id，存 status (active/completed), final_score。
> 6. **chat_messages**: 关联 session_id，存 role (user/assistant), content, metadata (JSON)。
> 7. **evaluation_reports**: 关联 session_id，存 scores (JSON: empathy/skill/safety), total_score, ai_feedback。
>
> 请确保 SQL 语法兼容 MySQL 8.0+。"
---
### 💡 为什么这样设计更适合 MVP？
1.  **开发快**：AI 写这种简单的 `CRUD` 表格准确率极高，不会因为外键约束报错。
2.  **修改灵活**：如果你想在 `evaluation_reports` 里加一个维度（如“沟通流畅度”），直接改 JSON 内容，不需要 `ALTER TABLE` 加列。
3.  **Python 友好**：Python 的 `dict` 和 `json` 模块处理 `scores` 字段非常自然，不需要ORM 复杂的映射。
