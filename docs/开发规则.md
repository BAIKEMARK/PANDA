## 1. 核心理念：6A 工作流
本项目强制执行 **6A 工作流**，确保 AI 生成代码的质量与可控性。
### 阶段 1: Align (对齐阶段)
-   **目标**：模糊需求 → 精确规范。
-   **产出**：`docs/任务名/ALIGNMENT_[任务名].md`，`docs/任务名/CONSENSUS_[任务名].md`。
-   **规则**：
    -   必须分析现有项目结构和依赖。
    -   必须识别歧义并主动询问，不可自作主张。
    -   明确任务边界和技术约束。
### 阶段 2: Architect (架构阶段)
-   **目标**：系统架构 → 模块设计 → 接口规范。
-   **产出**：`docs/任务名/DESIGN_[任务名].md`。
-   **规则**：
    -   绘制 Mermaid 架构图和数据流图。
    -   定义清晰的 MVC 分层接口。
    -   避免过度设计，确保与现有系统兼容。
### 阶段 3: Atomize (原子化阶段)
-   **目标**：拆分任务 → 明确依赖。
-   **产出**：`docs/任务名/TASK_[任务名].md`。
-   **规则**：
    -   任务拆解至原子级别（单一职责）。
    -   明确每个任务的输入契约、输出契约和依赖关系。
### 阶段 4: Approve (审批阶段)
-   **目标**：人工审查 → 执行确认。
-   **规则**：
    -   检查完整性、一致性、可行性。
    -   **只有通过审批后，才能开始编写代码。**
### 阶段 5: Automate (自动化执行)
-   **目标**：按节点执行 → 编写测试 → 实现代码。
-   **产出**：代码文件、单元测试、`docs/任务名/ACCEPTANCE_[任务名].md`。
-   **规则**：
    -   **测试优先**：先写测试，后写实现。
    -   **规范严格**：严格遵守代码风格（Black/Prettier）。
    -   **安全第一**：API Key 放入 `.env`，严禁硬编码。
    -   遇到阻塞立即中断并询问。
### 阶段 6: Assess (评估阶段)
-   **目标**：质量评估 → 文档更新 → 交付确认。
-   **产出**：`docs/任务名/FINAL_[任务名].md`，`docs/任务名/TODO_[任务名].md`。
-   **规则**：
    -   验收所有需求是否满足。
    -   生成清晰的操作指引 TODO 列表。

## 3. 后端 MVC 开发规范
### 3.1 编码标准
-   **风格**：严格遵循 **PEP 8**。
-   **工具**：`black` (格式化), `isort` (导入排序), `pylint` (静态检查)。
-   **类型**：所有函数必须包含 **Type Hints**。
### 3.2 分层职责
#### [Model] 数据层
-   **`models/`**: 定义 SQLAlchemy ORM 表结构，仅包含字段和关系。
-   **`schemas/`**: 定义 Pydantic 模型 (`*Create`, `*Update`, `*Response`)，这是 API 契约。
-   **`crud/`**: 封装基础 CRUD 操作，接收 `db` 和 ID，返回对象。
-   **`services/`**:
    -   包含复杂业务逻辑（如计算、LangGraph 调用、事务管理）。
    -   Controller 调用 Service，Service 调用 CRUD。
    -   **禁止**：直接访问 HTTP Request/Response 对象。
#### [Controller] 控制层
-   **`api/`**: FastAPI 路由文件。
-   **职责**：参数解析、依赖注入 (`Depends`)、调用 Service、返回 HTTP 响应。
-   **禁止**：
    -   直接编写 SQL 语句。
    -   包含复杂业务逻辑（应放入 Service）。
    -   直接返回 ORM 对象（必须通过 Pydantic 序列化）。
#### [View] 视图层
-   FastAPI 自动将 Pydantic 模型渲染为 JSON 响应。
### 3.3 LangGraph 规范
-   **定义**：State 必须使用 `TypedDict`。
-   **位置**：Graph 构建逻辑在 `app/graph/`，节点函数在 `services/`。
-   **异常**：节点内部必须捕获异常，防止工作流挂起。

## 4. 前端 MVC 开发规范
### 4.1 编码标准
-   **风格**：遵循 **Airbnb Style Guide**。
-   **工具**：`eslint`, `prettier`。
-   **类型**：`strict: true`，严禁使用 `any`，优先使用 `unknown`。
### 4.2 分层职责
#### [Model] 数据层
-   **`models/`**: 定义 TypeScript 接口。
-   **同步**：必须与后端 `schemas/` 保持一致。
#### [Controller] 控制层
-   **`controllers/`**: 封装 API 请求。
-   **职责**：处理 Token、错误码转换、数据格式化。
-   **禁止**：组件内部严禁直接写 `fetch`，必须调用 Controller。
#### [View] 视图层
-   **`app/` & `components/`**:
    -   负责 UI 渲染和事件处理。
    -   通过 Controller 获取数据。
    -   **禁止**：在 JSX 中包含复杂业务逻辑。

## 5. 数据库与安全规范
### 5.1 数据库
-   **ORM**：使用 **SQLAlchemy** (Async Mode)。
-   **迁移**：使用 **Alembic**，严禁手动修改生产库表结构。
-   **防注入**：严禁字符串拼接 SQL，必须使用 ORM 参数化。
### 5.2 安全
-   **密钥管理**：所有敏感信息（API Key, DB Password）必须存放在 `.env` 文件中。
-   **Git 忽略**：`.env` 文件**严禁**提交。
-   **认证**：所有接口（除登录外）必须通过依赖注入验证 Token。

## 6. 自动化规则执行
### 6.1 本地 Pre-commit Hooks
防止不规范代码进入本地仓库。
**文件：`.husky/pre-commit`**
```bash
#!/bin/sh
. "$(dirname "$0")/_/husky.sh"
# Frontend check
cd frontend && npx lint-staged
cd ..
# Backend check
cd backend && black --check . && isort --check-only .
cd ..
echo "✅ Pre-commit passed."
```
### 6.2 GitHub Actions CI
防止不规范代码进入远程主分支。
**文件：`.github/workflows/ci.yml`**
```yaml
name: CI Pipeline
on:
  push:
    branches: [ "main", "dev" ]
  pull_request:
    branches: [ "main", "dev" ]
jobs:
  backend-check:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ./backend
    steps:
      - uses: actions/checkout@v4
      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.10'
      - name: Install Dependencies
        run: |
          pip install black isort pytest
          pip install -r requirements.txt
      - name: Black Check
        run: black --check .
      - name: Import Sort Check
        run: isort --check-only .
      - name: Tests
        run: pytest
  frontend-check:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: ./frontend
    steps:
      - uses: actions/checkout@v4
      - name: Set up Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '18'
          cache: 'npm'
          cache-dependency-path: ./frontend/package-lock.json
      - name: Install Dependencies
        run: npm ci
      - name: ESLint
        run: npm run lint
      - name: Type Check
        run: npm run type-check
      - name: Tests
        run: npm run test
```

## 7. AI 交互与提示词配置
请将以下内容作为 AI 的 **System Prompt**，以确保其严格遵循上述规则：
```markdown
# Role
你是一位资深的软件架构师和全栈工程师，精通 React, Vite, FastAPI, Python 和 MVC 架构模式。你具备强大的上下文工程能力和规范驱动思维。
# Workflow: 6A Process
你必须严格遵循 6A 工作流来处理任何开发任务：
1. **Align**: 分析上下文，确认需求，生成 `ALIGNMENT` 和 `CONSENSUS` 文档。
2. **Architect**: 设计系统架构，定义 MVC 分层接口，绘制 Mermaid 图，生成 `DESIGN` 文档。
3. **Atomize**: 拆解原子任务，明确依赖关系，生成 `TASK` 文档。
4. **Approve**: 自我检查计划的完整性和可行性，等待用户审批。
5. **Automate**: 逐步执行任务，编写代码、测试，更新 `ACCEPTANCE` 文档。
6. **Assess**: 评估结果，生成 `FINAL` 和 `TODO` 文档。
# Coding Standards
1. **Architecture**:
   - Backend: `api` (Controller) -> `services` (Logic) -> `crud` (Data).
   - Frontend: `components` (View) -> `controllers` (API Client) -> `models` (Types).
   - 严禁分层越界（如 Controller 写 SQL，View 写 fetch）。
2. **Code Style**:
   - Python: PEP 8, Black, Type Hints.
   - TypeScript: Strict mode, NO `any`, Airbnb ESLint.
   - LangGraph: State must be TypedDict.
3. **Security & Env**:
   - API Keys must go into `.env` files. Never commit secrets.
   - Use ORM for DB operations (No raw SQL).
# Execution Rules
- 遇到不确定性或需要用户决策时，必须**中断执行并询问**。
- 生成代码时，明确标注文件路径。
- 保持代码精简易读，复用现有组件。
```
